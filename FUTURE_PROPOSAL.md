## 🔥 第一階段：近期高影響力優化（建議 1~2 個月內完成）

### 1.1 🔔 LINE Notify / LINE Bot 預約通知整合

**痛點**：使用者預約成功後沒有任何主動通知，容易忘記，或需手動回系統確認。  
**解法**：
1. 建立 LINE Official Account 或申請 LINE Notify Token（個人免費）。
2. 預約確認時推播：`📅 預約確認 | [張老師] 已成功預約 [禮堂] 2026/03/05 第三節`
3. **提前 1 小時**自動發送使用提醒：`⏰ 即將使用提醒 | 1小時後您有禮堂預約，請準時前往。`
4. 取消預約時推播：`❌ 預約已取消 | [禮堂] 2026/03/05 第三節 取消成功。`

**技術路徑**：Firebase Cloud Functions 監聽 Firestore 事件 → LINE Messaging API。  
**預估效益**：⭐⭐⭐⭐⭐ │ **技術難度**：⭐⭐ │ **預計開發時間**：3~5 天

> ⚠️ **2026 注意**：LINE Notify 已於 2025/3 停服，請直接採用 **LINE Messaging API（LINE Bot）**。

**痛點**：目前僅單一管理員帳號，無法區分「設備組」與「場地組」職責。  

| 角色 | 管理範圍 | 可操作動作 |

| **設備管理員** | IPAD 平板車（三~六年級）| 取消、查看借用紀錄 |

**技術路徑**：Firebase Auth Custom Claims + Firestore Rules 動態判讀。  
**預估效益**：⭐⭐⭐⭐⭐ │ **技術難度**：⭐⭐⭐ │ **預計開發時間**：5~7 天

### 🆕 1.5 🛡 智慧速率限流 v2（基於信任度動態調整）

**背景**：v2.38.6 雖將限制放寬至 30/小時、100/天，但仍是「一刀切」的硬限制。  
**痛點**：
- 行政組長批次預約會議室時可能仍會觸發
- 新使用者第一天就能預約 100 次，缺少漸進信任機制
- 惡意使用者只要分散時間就能繞過

**解法**：導入「信用積分制」與動態節流：

| 信用等級 | 條件 | 每小時 / 每天額度 |
| :---: | :--- | :---: |
| 🆕 新手 | 註冊 < 7 天 | 10 / 30 |
| ⭐ 一般 | 預約完成率 ≥ 70% | 30 / 100 (現行) |
| 💎 信賴 | 完成率 ≥ 90% & 無 30 天內取消 | 50 / 200 |
| 👑 行政 | 管理員授予的特殊角色 | 無上限（僅留稽核） |

**技術路徑**：
1. Firestore 新增 `userCreditScores` Collection，以 deviceId / userId 為 key。
2. 每次預約 / 取消時觸發 Cloud Function 重算分數。
3. 前端 `checkRateLimit()` 改為從 Firestore 讀取個人額度。

**預估效益**：⭐⭐⭐⭐⭐ │ **技術難度**：⭐⭐⭐ │ **預計開發時間**：5~7 天  
**附加效益**：徹底解決 v2.38 系列「不斷被迫放寬」的循環問題。

---

### 🆕 1.6 📲 Web Push 瀏覽器通知（PWA 進階）

**背景**：LINE Bot 雖完整，但仍需要老師加入官方帳號。Web Push 可在使用者點同意後直接推送，無需第三方平台。  
**解法**：
1. PWA 註冊 `Notification.requestPermission()` 取得授權。
2. Firebase Cloud Messaging (FCM) 取得 Push Token，存入 Firestore。
3. Cloud Function 在預約事件 / 提醒時間透過 FCM 推播。
4. 通知內容支援「點擊直接跳到該預約詳情頁」。

**預估效益**：⭐⭐⭐⭐ │ **技術難度**：⭐⭐⭐ │ **預計開發時間**：4~6 天  
**注意**：iOS Safari 需 16.4+ 才支援 Web Push，需保留 LINE Bot 作為備援。

### 🆕 1.7 ⚡ 一鍵重複上次預約（Quick Re-Book）

**痛點**：固定週課（例如每週三第三節在禮堂）每週都要重來一遍。  
**解法**：
1. 預約紀錄列表每筆右側加「🔁 再預約一次」按鈕。
2. 點擊後預填所有欄位（場地 / 節次 / 預約者 / 理由），日期自動跳到「下週同日」。
3. 衝突檢查同步觸發，若衝突直接顯示 AI 替代方案。

**預估效益**：⭐⭐⭐⭐ │ **技術難度**：⭐ │ **預計開發時間**：1~2 天  
**ROI 說明**：開發成本最低、立即提升日常效率，建議優先做。

---

### 🆕 1.8 🔍 完整稽核日誌（Audit Log）

**痛點**：強制刪除預約功能上線後，無法追溯「誰在什麼時候刪了什麼」。  
**解法**：
1. Firestore 新增 `auditLogs` Collection，欄位含 `action`、`targetId`、`operatorDeviceId`、`operatorRole`、`timestamp`、`payload (Before/After)`。
2. 每個寫入操作（建立 / 取消 / 強刪 / 範本套用）皆寫入一筆 log。
3. 管理員後台新增「稽核日誌」頁籤，支援篩選 + CSV 匯出。

**預估效益**：⭐⭐⭐⭐ │ **技術難度**：⭐⭐ │ **預計開發時間**：3~4 天  
**法遵效益**：如未來導入個資法 / 校務評鑑時可直接出具完整紀錄。

## 🔮 第三階段：長期願景（4 個月以上）

### 3.1 📱 PWA 強化 vs 原生 App

| 方案 | 優點 | 缺點 |
| :--- | :--- | :--- |
| **PWA 強化** | 無需上架、即時更新 | 功能受瀏覽器限制 |
| **React Native / Flutter** | 近原生體驗、可上 App Store | 開發成本高、審核週期長 |
| **Capacitor（Ionic）** | 現有 Web 程式碼直接打包 | 效能介於兩者之間 |

**建議**：先強化 PWA（Background Sync、Web Push API、離線預約佇列），若使用量足夠再考慮 Capacitor。

## 🛠 第四階段：開發品質與長期維運（持續推進）

> 此階段不是面向使用者的新功能，而是讓系統長久好維護、好擴充的基礎建設。

### 🆕 F.1 🧪 測試覆蓋率（Unit / E2E）

**現狀**：專案目前無自動化測試，每次更新需手動點選驗證。  
**建議**：
- **單元測試**：採用 **Vitest**，優先補上 `checkRateLimit()`、`detectConflict()`、AI 建議邏輯。
- **E2E 測試**：採用 **Playwright**，腳本錄製「預約 → 衝突 → 採用建議 → 取消」完整流程。
- **CI 整合**：GitHub Actions 每次 PR 自動跑測試，失敗禁止合併。

**預估效益**：⭐⭐⭐⭐⭐（防止回歸）│ **技術難度**：⭐⭐⭐ │ **預計工時**：10~14 天（含補測試案例）

### 🆕 F.2 📐 漸進式 TypeScript 遷移

## 💰 C 系列：成本與效能優化

### C.1 📦 Firestore 讀取量降低（IndexedDB 快取）

**痛點**：每次切換週次都重新查 Firestore，光是首頁載入就 50+ reads。  
**解法**：
1. 用 `idb` (IndexedDB wrapper) 在本地快取最近 4 週預約資料
2. 切換週次時優先讀本地，背景刷新
3. 收到 Firestore `onSnapshot` 變更通知時自動同步本地

**預期效益**：每月 Firestore 讀取量降低 60~70%，同時提升載入速度  
**預估效益**：⭐⭐⭐⭐⭐ │ **技術難度**：⭐⭐⭐ │ **預計開發時間**：3~4 天

### C.2 🪶 程式碼分割（Code Splitting）

**現狀**：`app.js` 單檔 3500+ 行，首次載入就要解析全部。  
**解法**：
1. 將以下功能改為「按需載入 (Dynamic Import)」：
   - 儀表板（佔 600+ 行）
   - 統計圖表（300+ 行）
   - CSV 匯出 / 月報表（200+ 行）
   - 批次預約日曆（300+ 行）
2. 使用 `import('./modules/dashboard.js').then(...)`

**預期效益**：首次載入 JS 從 ~150KB → ~80KB  
**預估效益**：⭐⭐⭐⭐ │ **技術難度**：⭐⭐⭐ │ **預計開發時間**：4~5 天

## 🔒 S 系列 — 安全與資料完整性

### S.1 🛡 重要操作雙因子確認 (2FA-Lite)

**痛點**:批次取消、強制刪除、改密碼等高風險操作只靠「輸入確認字」防誤觸,稍有疏忽即破壞性。  
**解法**:
1. 重要動作觸發時,系統發送 6 位數驗證碼至綁定 LINE Notify / Email / 手機
2. 使用者輸入後才執行
3. 驗證碼 5 分鐘失效
4. 管理員設定哪些動作要 2FA(白名單)

**對應教訓**:v2.41.8 的批次取消漏洞顯示「單純文字確認」不夠強  
**預估效益**:⭐⭐⭐⭐⭐ │ **技術難度**:⭐⭐⭐ │ **預計開發時間**:5~7 天

---

### S.2 📜 預約編輯歷史 (Edit Trail)

**痛點**:預約建立後若被改過(管理員代改、批次更新),完全無紀錄。  
**解法**:
1. Firestore 每筆 booking 增加 `editHistory: Array<{at, by, changes}>` 欄位
2. update 時自動 append 變更記錄(diff)
3. 預約詳情新增「📜 編輯歷史」分頁
4. 管理員可查任何預約的完整時間線

**範例**:
```
2026/03/15 14:23 by 王主任
  - reason: 「畢業典禮預演」 → 「畢業典禮彩排(含家長)」
2026/03/12 10:15 by 張老師(本人,初建立)
```

**預估效益**:⭐⭐⭐⭐ │ **技術難度**:⭐⭐⭐ │ **預計開發時間**:4~5 天

---

### S.3 🔍 自動化資安掃描 (Snyk / npm audit / OWASP)

**目標**:系統定期自動檢查相依套件 / 程式碼安全弱點。  
**解法**:
1. GitHub Actions 加入每週執行的 security scan
   ```yaml
   - uses: snyk/actions/node@master
   - run: npm audit --audit-level=moderate
   ```
2. 整合 [GitHub Dependabot](https://github.com/dependabot) 自動 PR 升級脆弱套件
3. 使用 [OWASP ZAP](https://www.zaproxy.org/) 掃描 GitHub Pages 站點

**預估效益**:⭐⭐⭐⭐ │ **技術難度**:⭐⭐ │ **預計開發時間**:2~3 天

---

### S.4 🔐 敏感欄位加密 (Field-Level Encryption)

**痛點**:目前 Firestore 純明文儲存。若 Firebase 帳號被盜,所有資料一覽無遺。  
**解法**:
1. 預約者姓名、預約理由 等個資欄位前端 AES 加密後存入
2. 解密金鑰存放於 Firebase Functions secrets(後端解密)
3. 對搜尋有影響:可用 [searchable encryption](https://en.wikipedia.org/wiki/Searchable_encryption) 或限制只能精確匹配

**預估效益**:⭐⭐⭐ │ **技術難度**:⭐⭐⭐⭐⭐ │ **預計開發時間**:2~3 週

---

### S.5 💧 列印/截圖浮水印

**情境**:有人截圖預約紀錄轉發到外面群組,如何追責?  
**解法**:
1. 每位使用者顯示資料時頁面加半透明斜紋浮水印(姓名 + 時間 + IP hash)
2. CSV 匯出時 PDF 渲染版本含完整浮水印
3. 設定可關閉(避免影響日常使用體驗)

**預估效益**:⭐⭐⭐ │ **技術難度**:⭐⭐ │ **預計開發時間**:2~3 天

### Q.3 🚧 React 風格錯誤邊界 (Error Boundaries)

**痛點**:單一函式 throw 整個 app 凍結(就像 v2.41.4 的 SW 錯誤)。  
**解法**:
1. 在主要 UI 區塊外包裝 try/catch
2. 統一 `safeExecute(fn, fallback)` wrapper
3. 失敗時顯示「⚠ 此區塊發生錯誤,點擊重試」而非整頁壞掉

## ⚡ P 系列 — 效能優化進階

### P.1 📇 Firestore 複合索引 + 查詢最佳化

**痛點**:目前 `executeAdvancedSearch` 一次查 180 天全部資料再前端過濾,Firestore 讀取量爆量。  
**解法**:
1. 在 `firestore.indexes.json` 加入 `date + room + periods` 複合索引
2. 改用 `where('room', '==', x).where('date', '>=', y)` server-side 過濾
3. 大幅降低 read 數,從 1000+/月降到 100~200

**預估效益**:⭐⭐⭐⭐⭐ │ **技術難度**:⭐⭐ │ **預計開發時間**:1~2 天

---

### P.2 🚀 關鍵預約資料 Preload

**情境**:首頁載入順序:HTML → CSS → JS → Firestore 查詢,使用者要等 800ms 才看到預約。  
**解法**:
1. 在 `<head>` 加入 `<link rel="preconnect">` 連 Firebase
2. JS 載入時即啟動 Firestore 查詢(不等 DOMContentLoaded)
3. 善用 IndexedDB 快取(配合 C.1)立即顯示

**預估效益**:⭐⭐⭐⭐ │ **技術難度**:⭐⭐ │ **預計開發時間**:1~2 天

---

### P.3 🖼 圖片改用 WebP / AVIF

**現狀**:favicon 是 PNG (~16KB)。  
**解法**:轉 SVG (1KB) 或 AVIF (2KB)。其他圖片(若有)同樣處理。  
**預估效益**:⭐⭐ │ **技術難度**:⭐ │ **預計開發時間**:半天

### P.4 🔤 中文字型 Subset

**解法**:用 [Subfont](https://github.com/Munter/subfont) 或 [Glyphhanger](https://github.com/zachleat/glyphhanger) 抽取實際用到的字符(約 1500 字),體積降至 ~200KB。

**預估效益**:⭐⭐⭐⭐ │ **技術難度**:⭐⭐⭐ │ **預計開發時間**:1~2 天
