# 資料保留政策與年度封存 SOP (DATA_RETENTION)

> 石門國小「禮堂&專科教室&IPAD平板車預約系統」資料生命週期管理規範
> 制定日期：2026-07-22（v2.51.0，L.4 提案落地）

---

## 📜 政策總則

| 項目 | 規定 |
| :--- | :--- |
| **保留年限** | 預約紀錄（`bookings`）至少保留 **3 個完整學年** |
| **封存時機** | 每學年結束（**7 月底**）執行一次「學期封存匯出」 |
| **清理條件** | 只有「已完成封存匯出 **且** 超過保留年限」的資料才可清理 |
| **清理方式** | 由管理員手動執行，**系統不做任何自動刪除** |
| **稽核日誌** | `audit_logs` 保留至少 1 學年，超過 5 萬筆時評估清理 |

## 🧠 為什麼是「先封存、後清理、不急著刪」？

2026-07-22 健檢結論（詳見 [PROGRESS.md](./PROGRESS.md)）：

1. **舊資料不影響效能** — 所有日常查詢都帶日期範圍 + `(room, date)` 複合索引，Firestore 查詢速度只跟「本次撈出的筆數」有關，跟資料庫總筆數無關。
2. **免費層空間充裕** — 每筆預約約 0.5KB，1 GiB 免費額度足以存放數十年的量。
3. **刪除不可逆** — 誤刪無法復原；保留則零風險。

因此清理的目的**不是效能，而是治理**（個資最小化保存原則）。不趕時間，寧可多留一年也不要誤刪。

## 🗓 年度 SOP（每年 7 月底，約 10 分鐘）

### 第 1 步：學期封存匯出（必做）

1. 以管理員身分登入系統。
2. 點「匯出 CSV」按鈕 → 開啟匯出彈窗。
3. 選擇「**本學年**」→ 按「**📦 學期封存匯出**」。
4. 取得兩個檔案：
   - `預約匯出_XXX學年(全學年)_<時間>.csv` — Excel 可直接開啟
   - `學期封存_XXX學年(全學年)_<時間>.json` — 含統計摘要（總筆數／各場地／逐月）+ 完整原始資料
5. 將兩檔存入學校雲端硬碟的「預約系統封存」資料夾（建議路徑：`預約系統封存/<學年>/`），並確認檔案可正常開啟。

### 第 2 步：檢查保留年限（必做）

- 打開 Firebase Console → Firestore → `bookings`，確認最舊資料的學年。
- **未滿 3 學年 → 到此結束，什麼都不用刪。**

### 第 3 步：清理過期資料（僅在超過保留年限且封存確認無誤時）

⚠️ **前置條件（缺一不可）**：
- [ ] 該學年已完成封存匯出（CSV + JSON 都在雲端硬碟）
- [ ] 已抽查封存檔內容（打開 CSV 核對筆數與 JSON summary.totalBookings 一致）
- [ ] 資料已超過 3 學年保留期限

確認後由管理員以腳本批次刪除該學年區間（`date >= 學年起` 且 `date <= 學年迄`）的 `bookings` 文件。刪除動作記入 `audit_logs`。

> 💡 目前（2026-07）資料庫僅累積約 1~2 學年資料，**最快也要 2028 年 7 月才會出現第一批可清理的資料**。在那之前每年只需做第 1、2 步。

## 📦 封存檔案格式說明

`學期封存_*.json` 結構：

```json
{
  "exportedAt": "匯出時間 (ISO 8601)",
  "system": "禮堂&專科教室&IPAD平板車預約系統",
  "range": { "label": "114學年(全學年)", "start": "2025/08/01", "end": "2026/07/31" },
  "summary": {
    "totalBookings": 615,
    "totalPeriods": 1834,
    "byRoom": { "禮堂": 210, "五年級IPAD車(28台)": 98 },
    "byMonth": { "2025/09": 88, "2025/10": 102 }
  },
  "bookings": [ { "id": "...", "date": "...", "room": "...", "periods": [], "booker": "...", "reason": "...", "createdAt": "...", "deviceId": "..." } ]
}
```

## 🔐 個資注意事項

- 封存檔含預約者姓名，屬個資 — **只能存放在學校管理的雲端硬碟**，不可上傳公開空間、不可放進 git repo。
- 提供他人查閱時，優先給 JSON 內的 `summary` 統計摘要，避免流出完整名冊。
